## הכלל המאחד: שלישיית בידוד הכיוון

המודל המנטלי — שלושה מאקרו, כל אחד לכיוון אחר:

| מאקרו | מה הוא | מתי |
| :---- | :---- | :---- |
| `\h{...}`    | אי **עברית** (RTL) | טקסט עברי בתוך הקשר LTR — צמתי TikZ, ה-island ה-`tikzpic`/`qcirc` |
| `\en{...}`   | בידוד **לטינית** (LTR) | טקסט לטיני / **סוגריים** בתוך הקשר עברי (RTL) — שומר על הסדר והסוגריים |
| `\code{...}` | `\en` + מונוספייס | **קוד** בתוך עברית או בתוך `frametitle` |

הגדרתם (כבר בפרמבל):

```latex
\newcommand{\h}[1]{\foreignlanguage{hebrew}{#1}}
\newcommand{\en}[1]{\foreignlanguage{english}{#1}}
\newcommand{\code}[1]{\foreignlanguage{english}{\texttt{#1}}}
```

> **חוק הברזל:** לעולם אל תעטוף לטינית ב-`\h{}` (זה המאקרו ל**עברית**). ולעולם אל
> תשתמש באף אחד מהשלושה בתוך **מצב מתמטי** — ראה הסעיף האחרון.

## ★★ הבאג המרכזי: לטינית בתוך `\h{}` בתוך צומת TikZ יוצאת הפוכה

זה הבאג שעלה הכי הרבה זמן בפרויקט הזה, והוא **מנוגד** לכלל 4 של `recipe.md`:

- **כלל 4** (recipe): אל תכתוב **עברית** בתוך מתמטיקה / תווית דיאגרמה — תצא הפוכה.
- **כאן:** אל תכתוב **לטינית** בתוך `\h{}` בתוך **צומת TikZ** — היא תצא הפוכה.

### התסמין

צומת TikZ שכתבת בה `\node {\h{מעבד (CPU)}}` או `\node {\h{Datapath}}` מציירת:

```
מעבד (CPU)   ->  מעבד )UPC(      (סוגריים שוקפו, CPU התהפך)
Datapath     ->  htapataD        (כל האותיות הלטיניות התהפכו אות-אות)
(C, Java)    ->  )avaJ ,C(
```

### למה זה קורה (והבחנה קריטית)

`bidi=basic` מסדר נכון ערבוב עברית/לטינית **בטקסט רגיל ובטבלאות** — שם אלגוריתם
ה-bidi רץ והריצה הלטינית נשארת LTR. אבל **בתוך צומת TikZ** הטקסט מוקלד בקופסה
שה-bidi לא מעבד באותו אופן, ולכן כיוון הכתיבה RTL חל ברמת הגליף בלי ההגנה של ריצת
ה-LTR — והלטינית יוצאת הפוכה.

> **המסקנה המעשית:** טבלאות `tabular`/`longtable` עם `\h{מיידי (Immediate)}` —
> **תקינות** (ה-bidi מעבד אותן). פסקאות רגילות — תקינות. **רק צמתי TikZ נשברים.**
> אם משהו נראה הפוך, שאל קודם: זה בתוך `tikzpicture`?

### התיקון — לפי סוג הצומת

| סוג הצומת | מה לעשות |
| :---- | :---- |
| **עברית בלבד** | `\h{...}` — עובד מצוין |
| **לטינית/קוד בלבד** | טקסט חשוף בתוך ה-island ה-`tikzpic` (בלי `\h`) — LTR נכון |
| **מעורב** | עטוף הכול ב-`\h{...}` (בסיס RTL), ובַדד כל מקטע לטיני/סוגריים ב-`\en{}`/`\code{}` |

```latex
% ❌ שגוי — לטינית בתוך \h בצומת:
\node {\h{מעבד (CPU)}};
\node {\h{Datapath}};
\node {\h{מטמון (SRAM)}};

% ✅ נכון — לטינית מבודדת:
\node {\h{מעבד \en{(CPU)}}};      % עברית RTL בבסיס, (CPU) כ-LTR
\node {\en{Datapath}};            % לטינית טהורה — בלי \h
\node {\h{מטמון \en{(SRAM)}}};
```

> **סוגריים סביב עברית הם בסדר.** `\h{זמן (מחזורים)}` — הסוגריים שוקפים באופן עקבי
> בצומת אחיד-RTL וקוראים נכון. **רק לטינית** בתוך `\h{}` צריכה בידוד. (`$\to$`,
> `$\rightarrow$` וכו' הם פקודות מתמטיות — לא טקסט לטיני — ולא צריכים בידוד.)

## ★★ כיוון חץ־זרימה בפרוזה עברית: `\leftarrow`, לא `\to`

ה-bidi **לא משקף** גליפי חצים. לכן `\to`/`\rightarrow` (←→ פוינטר ימינה →) בתוך
פרוזה עברית RTL מצביעים **לאחור**, נגד כיוון הקריאה. חץ־זרימה בין שלבים בעברית
("שיוך → עדכון → חזרה") צריך להצביע **שמאלה**, בכיוון הקריאה והזרימה:

```latex
שיוך $\to$ עדכון $\to$ חזרה          % ❌ → החצים מצביעים ימינה (לאחור)
שיוך $\leftarrow$ עדכון $\leftarrow$ חזרה   % ✅ ← בכיוון הקריאה (שלב → הבא)
```

אותו דבר ל**גרירה לוגית inline** בפרוזה: `אי־תלות $\Rightarrow$ Cov=0` מצביע
לאחור; השתמש ב-`$\Leftarrow$` כך שהחץ מצביע מההנחה (ימין) למסקנה (שמאל).

**מה **לא** לשנות (מתמטיקה/קוד טהורים, נשאר `\to`):** גבול `$\lambda\to\infty$`,
מיפוי `$f:A\to B$`, `$x\mapsto y$`, התכנסות `$\xrightarrow{d}$`, שיוך נקודה
`$1\to\mu_1$`, וחצים בין אופרנדים ש**כולם** מתמטיקה/קוד (`\code{lw}\to\code{add}`,
`\mathrm{EX}\to\mathrm{EX}`, `$[b_{n-1}\dots b_0]\mapsto[\dots]$`) — כל אלה נקראים
שמאל-לימין בתוך עצמם, והחץ נכון.

**אזהרה — משוואת display אינה LTR אוטומטית.** ההנחה ש״משוואות מסודרות LTR ממילא
ולכן `\Rightarrow` תמיד נכון״ **שגויה**: משוואה שמערבת בלוקי `\text{עברית}`
**כאופרנדים** (לא כתוויות `_{\text{...}}` של `\underbrace`/אינדקס) נקראת **RTL**,
ואז `\Rightarrow`/`\to` שבתוכה **מצביעים לאחור**. הדוגמה ששברה — `\eqref{}` של חריגה:
`\text{חריגה ב־}EX \Rightarrow \text{בטל}…` (המקדים יצא מימין, המסקנה משמאל, וה-
`\Rightarrow` הצביע ימינה = לאחור). זה חל גם כש-`\text` העברי רק **באחד** האגפים
(`\code{div} \Rightarrow \code{Lo}=\text{מנה}`) או רק ב**תחילת** המשוואה
(`\text{אם } … \Rightarrow …`) — די בעברית אחת כדי להפוך את כל המשוואה.

**הפתרון המומלץ (הכי קריא): נסח מחדש בלי חץ כיווני** — נקודתיים או מילת־קישור עברית:
`\text{חריגה ב־}EX:\ \text{בטל}…`, `\text{אם } … \ \text{אז}\ …`,
`\code{div}:\ \code{Lo}=\text{מנה}`. חלופה: `\Leftarrow`/`\leftarrow` (מצביע שמאלה,
מההנחה שמימין למסקנה שמשמאל). השאר `\to`/`\Rightarrow` **רק** כשכל האופרנדים
מתמטיקה/קוד. הכלל המעודכן: **חץ שאחד מאופרנדיו עברי (פרוזה או `\text` במשוואה) →
`\leftarrow`/`\Leftarrow` או ניסוח מחדש ללא חץ; חץ בין אופרנדים מתמטיים/קוד בלבד →
נשאר `\to`/`\Rightarrow`.** `check_bidi_figures.py` מסמן `HEBREW-FLOW-ARROW`
(חץ בודד ב-`$..$` בין פרוזה עברית) ו-`HEBREW-EQUATION-ARROW` (חץ במשוואה עם אופרנד `\text` עברי).

## ★★ אל תשים **משפט עברי** בתוך `equation`/`align`/`\[..\]` — הוא נקרא הפוך (LTR)

מתמטיקת display היא **LTR** (חל על `\begin{equation}`, `align`, **וגם `\[..\]`**).
הניואנס המדויק: בלוק `\text{...}` **בודד** (אפילו רב-מילים, למשל `\text{(מבחן דו־צדדי)}`
או תווית `\underbrace{}_{\text{תת־התאמה}}`) נקרא RTL **תקין** — הוא יחידה אחת. הבעיה
היא **משפט שמפוצל לכמה בלוקי `\text` עם מתמטיקה ביניהם**: הבלוקים מסודרים שמאל-לימין,
ולכן המשפט יוצא הפוך. דוגמה אמיתית שנשברה — רווח סמך:

```latex
\[ \bar X \pm z_{0.975}\tfrac{\sigma}{\sqrt n}              % ❌ משפט מפוצל:
   \quad\text{(או }t_{n-1}\tfrac{s}{\sqrt n}\text{ כש־}\sigma\text{ לא ידועה).} \]
% ה"(או … כש־σ לא ידועה)" יוצא משובש כי הוא ב-3 בלוקי \text סביב מתמטיקה.
% ✅ מתמטיקה טהורה ב-display, וההסבר העברי כפרוזת RTL בין השורות:
\[ \bar X \pm z_{0.975}\,\tfrac{\sigma}{\sqrt n}. \]
כאשר $\sigma$ אינה ידועה, מחליפים ב־$s$ ומשתמשים בהתפלגות $t$:
\[ \bar X \pm t_{n-1,0.975}\,\tfrac{s}{\sqrt n}. \]
```

**הכלל:** טענה/יחס **מילולי בעברית** (או משפט מפוצל לבלוקי `\text`) → שורת
`\begin{center}…\end{center}` / פרוזת RTL עם סמלים inline, ו-`\leftarrow`/`\Leftarrow`/
מילים ("גורר", "כאשר", "ככל ש"). **מתמטיקה טהורה** (`\hat\beta=(X^\top X)^{-1}X^\top y`,
`\Loss(f_{-j},D_j)`) → נשארת ב-`equation`/`\[..\]`; דחוף תיאורים עבריים לפרוזה אחרי
הנוסחה. **תווית בודדת** (`\text{עוצמה}=1-\beta`, תווית `\underbrace`) — תקינה. בדוק
בעין: אם מילה עברית ראשונה קופצת לשמאל, או סוגריים/מילים מתערבבים — זה הבאג.

**תסמין נוסף — רווח־גבול נעלם בכריך עברית־לטינית־עברית.** מונח לטיני בודד הלכוד בין
שתי מילים עבריות בתוך משוואה דרך אטומי `\text`/`\mathrm` נפרדים מאבד את הרווח שלפניו:
`\text{וקבע } \mathrm{PC} \text{ לכתובת}` נדפס **`וקבעPC`** (ה-bidi בולע את הרווח בגבול
RTL↔LTR; הרווח שבתוך `\text{... }` לא שורד שם). זה קרה למשפט חריגה — `\eqref{}` 29.1.
**הפתרון:** אם צריך ״משוואה״ שהיא בעצם **משפט עברי עם תוויות לטיניות** (בלי אופרטור
מתמטי אמיתי), כתוב אותה כ**בלוק `\text` אחד** עם איי `\en` ללטינית (מצב טקסט →
רווחים אמינים, בדיוק כמו בפרוזה), לא כשרשור `\text{עברית } \mathrm{לטינית} \text{ עברית}`:
```latex
% ❌ אטומים נפרדים — הרווח לפני PC נבלע: "וקבעPC"
\text{חריגה ב־}\mathrm{EX}:\quad \text{בטל } \mathrm{IF} \text{, וקבע } \mathrm{PC} \text{ לכתובת המטפל}.
% ✅ בלוק \text יחיד עם איי \en (רווחים אמינים)
\text{חריגה ב־\en{EX:} בטל \en{IF, ID, EX}, וקבע \en{PC} לכתובת המטפל.}
```
שתי מלכודות בתוך הבלוק האחד: **מזג לטינית סמוכה** לאי `\en` יחיד (`\en{IF, ID, EX}`,
אחרת `ADJACENT-LTR-ISLANDS` הופך ל־"EX, ID, IF"), ו**נקודתיים נגררת אחרי אי `\en`
שים בתוך האי** (`\en{EX:}`, אחרת `MATH-TEXT-EN-COLON` מיתמת אותה ל־":EX").

**בונוס — קירוב בפרוזה:** `$\approx 80\%$` בתוך פרוזה עברית *קריא נכון* (≈ הוא אי LTR
זעיר), אבל נראה זר ולא-אידיומטי. בפרוזה עדיף עברית: **`כ־$80\%$`** ("כ־") או "בערך".
שמור את `\approx` ל**חישובים/משוואות** (`=\frac{\dots}{\dots}\approx 0.083`), שם הוא תקני.
אותו עיקרון ל-`<`/`>`/`\le` מילוליים: בפרוזה עדיף מילים ("קטן מ־", "לכל היותר").

## ★★ כותרות קופסה עם קוד: `\code{}` ולא `\texttt{}`

> **תבניות קופסה מוכנות להעתקה:** לפני שבונים קופסה מאפס — פתחו את `assets/example_boxes.tex`
> (גלריה של כל שש הקופסאות: `defbox`/`thmbox`/`notebox`/`exbox`/`warnbox`/`keybox`, כל אחת
> מוערת בתפקידה). **בחרו קופסה לפי תפקיד, לא צבע**; הכותרת נמסרת פר-מופע ב-`[frametitle={...}]`.
> הגלריה מדגימה גם את שני הדפוסים: **צעד שורה-לכל-פריט** (`\\`) ב-`keybox`, ו-`nobreak=true`
> לקופסת-ייחוס קצרה.

`frametitle` של קופסה הוא הקשר RTL. אם תכתוב בו קוד C חשוף עם `\texttt{}`:

```latex
\begin{exbox}[frametitle={דוגמה — \texttt{if (i==j) f=g+h; else f=g-h;}}]
```

ה-bidi מעיף את הנקודה-פסיק הסוגרת **להתחלה** ומבלבל את הפיסוק:

```
דוגמה — if (i==j) f=g+h; else f=g-h;   ->   ;if (i==j) f=g+h; else f=g-h — דוגמה
```

הסיבה: `\texttt{}` קובע מונוספייס אבל **לא** קובע כיוון. הקוד נשאר בהקשר RTL,
והפיסוק ה"ניטרלי" (`;`, `()`, `==`) בגבולות נפתר לצד ה-RTL.

**התיקון — `\code{}` במקום `\texttt{}` בכל כותרת:**

```latex
\begin{exbox}[frametitle={דוגמה — \code{if (i==j) f=g+h; else f=g-h;}}]
```

`\code{}` מוסיף `\foreignlanguage{english}` (LTR), והקוד כולו — כולל הפיסוק —
יוצא משמאל לימין נכון, עם הנקודה-פסיק בסוף. (מזהים קצרים כמו `\texttt{srl}` בלי
פיסוק יוצאים בסדר גם בלי בידוד, אבל המרה ל-`\code{}` בכל הכותרות עקבית ובטוחה.)

המרה גורפת לכל הכותרות במסמך (פייתון):

```python
import re, glob
def fix(s):
    out=[]; i=0; key='frametitle={'
    while i < len(s):
        j=s.find(key,i)
        if j<0: out.append(s[i:]); break
        out.append(s[i:j+len(key)]); k=j+len(key); d=1
        while k<len(s) and d>0:
            if s[k]=='{':d+=1
            elif s[k]=='}':d-=1
            if d==0:break
            k+=1
        out.append(s[j+len(key):k].replace(r'\texttt{', r'\code{'))
        out.append(s[k] if k<len(s) else ''); i=k+1
    return ''.join(out)
for fn in glob.glob('*.tex'):
    s = open(fn, encoding='utf-8').read()         # read FULLY first — open(fn,'w') truncates
    open(fn, 'w', encoding='utf-8').write(fix(s))  # immediately, so never read+write in one expr
```

> אותו עיקרון חל על **שורות פרוזה** עם קוד עתיר-פיסוק: עטוף ב-`\code{}`/`\en{}`.
> מילה לטינית בודדת בפרוזה — ה-fallback מספיק; קוד עם `;()==` — בַדד.

## ★★ מזהה לטיני צמוד למתמטיקה בפרוזה — נדבק והופך

זהו הקרוב של באג הכותרת (Rule 8), אבל **בגוף הטקסט**, והוא עלה בפועל בקופסאות
הליכה-לפי-הוראה (הצבות אותות בקרה). הדפוס השבור:

```latex
\texttt{ALUOp}$=10$, \texttt{RegDst}$=1$, \texttt{RegWrite}$=1$.
```

מרנדר בהקשר RTL כ:

```
10ALUOp ,= 1RegDst ,= 1RegWrite      (המספר נדבק לשם, ה-= מרחף, הסדר מתערבב)
```

**הסיבה:** `\texttt{}` קובע מונוספייס אבל **לא כיוון**. \texttt{ALUOp} היא ריצת לטינית
LTR לא-מבודדת, ו-`$=10$` היא מתמטיקה (LTR). שתי ריצות LTR נפרדות **צמודות** בהקשר
RTL — ה-bidi מסדר אותן בנפרד והן מתנגשות. אותו דבר קורה לביטוי `\texttt{\$rs}$+$\texttt{\$rt}`
(נדבק ל-`$+rs$`).

**התיקון — עטוף את כל ההצבה כיחידת LTR אחת עם `\code{}`:**

```latex
% ❌ שבור — שם ב-\texttt צמוד למתמטיקה נפרדת:
\texttt{ALUOp}$=10$
\texttt{\$rs}$+$\texttt{\$rt}

% ✅ נכון — ההצבה/הביטוי כולו יחידת LTR אחת:
\code{ALUOp=10}
\code{\$rs + \$rt}
```

הכלל הכללי: **מזהה לטיני + אופרטור/ערך** (`name=val`, `a+b`, `x<-y`) שייכים ל-`\code{}`
**אחד**, לא ל-`\texttt{}` שמפוצל ממתמטיקה. ה-`=`/`+`/הספרות נכנסים פנימה ל-LTR ולא
מתנגשים.

### סוגריים/מרובעים/`$` בתוך `\texttt{}` (מיעון MIPS) — תוקן מבנית ב-3.7

**ההיקף האמיתי (נמדד):** לא רק סוגריים. בתוך `\texttt{}` בהקשר עברי **מתהפכים/נודדים**:
סוגריים עגולים `0($s0)`→`(s0$)0`, **סוגריים מרובעים** `arr[i]`→`[arr[i`, ו**`$`** לבדו
`$rs`→`rs$` (וזה ב**כל** רגיסטר MIPS!). תקינים בתוך `\texttt{}`: `; : / = + - < >`.
הסיבה: `\texttt{}` קובע גופן מונוספייס אבל **לא כיוון**, אז כל תו נייטרלי בפנים יושב בריצת
ה-RTL ומתערבב.

**התיקון המבני (3.7):** ה-preamble מגדיר מחדש את `\texttt` כך שיבדד את תוכנו ל-LTR
(עטוף ב-`\ifmmode` כדי לא לשבור מתמטיקה). מאז — **`\texttt{}` רגיל בטוח אוטומטית**, ואין
צורך לזכור `\code{}`:

```latex
% ✅ עכשיו תקין מאליו (3.7) — בלי \code:
בקטע (\texttt{lw \$t0,0(\$s0)}) ואחריו      % -> (lw $t0,0($s0))
\texttt{arr[i]}  \texttt{\$rs}  \texttt{g(f(x))}   % כולם תקינים
```

`\code{}` נשאר ככינוי מפורש (זהה ל-`\texttt` עכשיו). **חשוב:** `\texttt{עברית}` עדיין tofu —
אבל זה היה תמיד כך (גופן המונוספייס חסר עברית), לא רגרסיה. עברית שייכת ל-`\h{}`/טקסט רגיל,
לעולם לא ל-`\texttt{}`. לחזרה לגרסת `\texttt` הסטנדרטית — מחק את הבלוק ב-preamble.

### בונוס: רב-שלבי → שורה לכל שלב, כמו משוואות

תיאור רב-שלבי (שלבי צנרת, רשימת אותות בקרה) שנדחס למשפט אחד עם `\;` בין השלבים —
גם קשה לקריאה וגם מגדיל את שטח ההתנגשות של ה-bidi. עדיף **שורה לכל שלב** (עם `\\`),
ואת ההצבות בשורה ייעודית כיחידות `\code{}` נקיות — קריא כמו משוואות:

```latex
\begin{keybox}[frametitle={\code{add \$rd,\$rs,\$rt} (R-type)}]
\textbf{\en{IF:}} שלוף הוראה, \code{PC = PC + 4}.\\
\textbf{\en{ID:}} קרא \code{\$rs}, \code{\$rt} מקובץ הרגיסטרים.\\
\textbf{\en{EX:}} ה-ALU מחשב \code{\$rs + \$rt}.\\
\textbf{\en{MEM:}} לא בשימוש.\\
\textbf{\en{WB:}} כתוב את התוצאה ל-\code{\$rd}.\\[3pt]
\textbf{אותות בקרה:} \code{RegDst=1}, \code{ALUOp=10}, \code{RegWrite=1}.
\end{keybox}
```

> **תווית שלב:** `\textbf{\en{IF:}}` — השם **והנקודתיים יחד** בתוך `\en{}` (אי LTR
> אחד). כך התווית נקראת "IF:". אם משאירים את הנקודתיים **מחוץ** ל-`\en{}`
> (`\textbf{\en{IF}:}`) הן מתייתמות לצד הלא נכון (":IF" לקורא RTL) — **מאומת
> אמפירית בפרוזה**, ובתוך `align`/`\text{}` זה ודאי. ה-checker מסמן `\text{\en{X}:}`
> כ-`MATH-TEXT-EN-COLON`.

### עוד שתי וריאציות של אותו עיקרון (יחידת LTR אחת)

**(א) קוד מופרד במפריד עם `\$` — אַחֵד ל-`\code{}` אחד.** `\texttt{\$imm1}/\texttt{\$imm2}`
(שני `\texttt{}` עם `/` ביניהם) — סימן ה-`\$` (מחלקת bidi מיוחדת) **נדבק במקום הלא
נכון**. שתי ריצות LTR נפרדות עם מפריד ניטרלי, וה-`\$` מתערבב.

```latex
\texttt{\$imm1}/\texttt{\$imm2}      % ❌ → הדולר במקום הלא נכון
\texttt{\$zero}/\texttt{\$imm1}/\texttt{\$imm2}   % ❌ (גם שלישייה)
\code{\$imm1/\$imm2}                 % ✅ יחידה אחת
\code{\$zero/\$imm1/\$imm2}          % ✅
```

> **הבחנה (עודכן v3.10):** פעם נכתב כאן שמפריד בין שתי ריצות לטיניות של **אותיות
> בלבד** (`\texttt{lw}/\texttt{sw}`) הוא תקין מכוח כלל N1 של bidi. **זה כבר לא נכון:**
> מאז v3.7 `\texttt` עטוף בבידוד LTR, כך ש-`\texttt{lw}` ו-`\texttt{sw}` נמצאים ברמות
> bidi נפרדות וה-N1 לא מגשר ביניהן — התוצאה **מתהפכת** ל-`sw/lw` (מאומת אמפירית).
> **לכן: תמיד אַחֵד** — `\texttt{lw/sw}`, `\code{add/sub}` (ריצה אחת), לא
> `\texttt{lw}/\texttt{sw}`. זה תקף לכל מפריד פסיק/לוכסן בין איי `\en`/`\code`/`\texttt`.
> ה-checker מסמן זאת כ-`ADJACENT-LTR-ISLANDS`.

**(ב) סוגריים שלפניהם לטינית בתוך עברית — עטוף ב-`\en{}`.** זה הדפוס הכי נפוץ
ומטעה, אז הנה הכלל המדויק (נבדק אמפירית):

- **מתהפך** (`(` נראה כמו `)`): כשמיד **לפני** ה-`(` יש אסימון **לטיני** (ראשי-תיבות,
  מילה, או אות בודדת — גם בתוך `\textbf{}`) בתוך טקסט עברי. לדוגמה:
  `CPI (Cycles Per Instruction)`, כותרת `\subsection{פורמט R (Register)}`,
  `\textbf{SRAM} (Static RAM)`, `ה-ALU (Arithmetic Logic Unit)`.
- **תקין** (אין צורך לגעת): כשלפני ה-`(` יש **עברית** — וזה הרוב המוחלט:
  `צנרת (pipelining)`, `מקומיות (Locality)`, `סיכונים (Hazards)`. וגם תאים/ריצות
  **טהורי-לטינית** (כל התא לטיני, בלי עברית סביב): `Direct Memory Access (DMA)`
  בטבלת מונחים, `K (Kilo)` בטבלה — שם כל הריצה LTR והסוגריים ממילא נכונים.

**התיקון:** עטוף את האסימון-הלטיני+הסוגריים ב-`\en{}` (או רק את הסוגריים):

```latex
CPI (Cycles Per Instruction)          % ❌ מתהפך (CPI לטיני לפני הסוגר)
CPI \en{(Cycles Per Instruction)}     % ✅
\subsection{פורמט \en{R (Register)}}  % ✅ כותרת
\textbf{SRAM} \en{(Static RAM)}        % ✅
צנרת (pipelining)                      % ✅ כבר תקין (עברית לפני) — אל תיגע
```

**(ג) רשימת פריטים מרובים בסוגריים — מתהפכת בסדר.** `(\code{addi}, \code{lw}, \code{sw})`
— כל `\code{}` הוא אי-LTR נפרד, ובהקשר RTL הם מסודרים מימין-לשמאל, אז הפריטים
**מופיעים הפוך** ("`sw, lw, addi`"). עטוף את כל הרשימה ב-`\en{}` (עם `\texttt{}`
בפנים) כך שתהיה ריצת LTR אחת והפריטים בסדר:

```latex
לקפיצות (\code{j}, \code{jal}):                         % ❌ → (jal, j)
לקפיצות \en{(\texttt{j}, \texttt{jal})}:                % ✅ (j, jal)
% רשימה עם עברית בתוכה — עטוף רק את חלק הקוד:
(\en{\texttt{add}, \texttt{sub}}, היסטים)               % ✅
```

(אם כל התוכן מונוספייס — `\code{(...)}` עוד יותר פשוט. בתוך `\en{}` השתמש ב-`\texttt{}`
לחלקים שצריכים מונוספייס; שאר הטקסט והמספרים בפונט רגיל.)

**(ד) הכלל הכללי — וזה לא רק קוד (v3.10).** כל מה שלמעלה הוא מקרה פרטי של עיקרון
אחד: **שני איי-LTR (`\en`/`\code`/`\texttt`/מתמטיקה) שמופרדים רק בתו ניטרלי
(פסיק, לוכסן, נקודתיים) מתהפכים בסדר ב-RTL.** זה תופס באותה מידה על **אנגלית
רגילה**, לא רק על קוד — וזה הכי קל לפספס כי זה נראה תמים:

```latex
\en{Griffiths}, \en{Thomson}, \en{PDG}    % ❌ → "PDG, Thomson, Griffiths"
\en{Griffiths, Thomson, PDG}              % ✅
\en{BaBar}, \en{Belle}                    % ❌ → "Belle, BaBar"
\en{BaBar, Belle}                         % ✅
\en{keV}, \en{MeV}                         % ❌ → "MeV, keV"
\en{keV, MeV}                             % ✅
\en{Breit-Wigner} / \en{Lorentzian}       % ❌ → "Lorentzian / Breit-Wigner"
\en{Breit-Wigner / Lorentzian}            % ✅
```

**מקרה הנקודתיים-מתמטיקה** (תווית אנגלית, נקודתיים, ואז `$...$`) — אותו באג, אבל
הנקודתיים מתייתמות: `\en{s-channel}: $q=p_1+p_2$` יוצא `q=p_1+p_2 :s-channel`,
ו-`(\en{detector-stable}: $p,n,e$)` שם את הרשימה לפני התווית. אַחֵד הכל ל-`\en{}` אחד:

```latex
\en{s-channel}: $q=p_1+p_2$               % ❌ → ":s-channel" יתום
\en{s-channel: $q=p_1+p_2$}               % ✅ "s-channel: q=p_1+p_2"
(\en{detector-stable}: $p,n,e$)           % ❌
\en{(detector-stable: $p,n,e$)}           % ✅
```

> **מה כן בטוח (אל תאחד סתם):** **רווח** רגיל בין אי לבין מתמטיקה (`\en{word} $x$`)
> **תקין** — רק צמידות-אפס (`\en{word}$x$`) או נקודתיים גורמות לבעיה. **נקודה**
> (`\en{First}. \en{Second}`) **לא** מתהפכת. רק **פסיק/לוכסן** בין איים, ו**נקודתיים
> לפני מתמטיקה**, הם הטריגרים — הכל מאומת אמפירית. ה-checker מסמן
> `ADJACENT-LTR-ISLANDS` (פסיק/לוכסן) ו-`ISLAND-NEXT-TO-MATH` (וריאנט הנקודתיים).

## ★★ רשימות קוד בתוך קופסאות: תיקון החריגה מהמסגרת

הבעיה: רשימת `lstlisting` בתוך קופסת `mdframed` מחשבת את השוליים שלה ביחס לשולי
**העמוד**, לא לשולי הקופסה. התוצאה: הפס הצדדי ומספרי השורות **בולטים שמאלה** מעבר
לגבול הקופסה ("הקוד לא נמצא בתוך המרובע").

**התיקון — שלושה מרכיבים בסגנון הרשימה:**

```latex
\lstdefinestyle{asm}{
  ...
  xleftmargin=2.2em,        % הזחה קטנה — מקום למספרי השורות מבפנים
  framexleftmargin=0pt,     % ★ הפס לא מתרחב שמאלה מעבר לקוד
  framexrightmargin=0pt,
  resetmargins=true,        % ★ שוליים נמדדים מהקופסה העוטפת, לא מהעמוד
  ...
}
```

ובנוסף — תן לקופסאות ריווח פנימי כדי שיהיה מקום:

```latex
\newmdenv[..., innerleftmargin=9pt,innerrightmargin=9pt, ...]{exbox}
```

`resetmargins=true` לבדו **לא מספיק** — חייבים גם `framexleftmargin=0pt`, אחרת הפס
עדיין מתרחב שמאלה אל מחוץ לקופסה. שילוב השלושה: הפס יושב בקצה השמאלי של הקוד בתוך
הקופסה, ומספרי השורות בין הפס לגבול. אומת בעין.

עטוף תמיד את הרשימה ב-island של LTR (קוד הוא LTR):

```latex
\newenvironment{asmblock}{\par\begin{otherlanguage}{english}}{\end{otherlanguage}\par}
...
\begin{asmblock}
\begin{lstlisting}[style=asm]
Loop: sll  $t1, $s3, 2     # t1 = i*4
      ...
\end{lstlisting}
\end{asmblock}
```

> **מלכודת נלווית — `\\` מוביל:** `\\` מיד אחרי `\begin{otherlanguage}` נותן
> `! LaTeX Error: There's no line here to end.` הסר אותו. (קורה כשמתחילים גוש קוד
> ב-`\\`.)

## ★★ גרשיים עבריים: ״ ולא ``...''

ב-LaTeX, `` `` `` (שני גרשים הפוכים) מפיק מרכאה **פותחת** מסולסלת — “ — בצורת "66",
ו-`''` מפיק מרכאה **סוגרת** — ” — בצורת "99". אלה **שני גליפים שונים בכוונה**,
נכונים לאנגלית אבל **שגויים לעברית**: בעברית משתמשים ב**גרשיים ״** (U+05F4) —
סימן סימטרי יחיד, זהה משני הצדדים. בכיוון RTL חוסר הסימטריה אף בולט יותר.

**התיקון — החלף את כל `` `` ``/`''` בגרש עברי ״:**

```python
import glob
G='\u05f4'  # ״
for fn in glob.glob('*.tex'):
    s=open(fn,encoding='utf-8').read()
    open(fn,'w',encoding='utf-8').write(s.replace('``',G).replace("''",G))
```

בונוס: גם **קיצורים עבריים** במסמך (ע״י, סה״כ, כנ״ל) השתמשו לרוב ב-`''` — ההמרה
מתקנת גם אותם לצורה הנכונה. Frank Ruhl Libre מכיל את הגליף ״ (אומת: 0 תווים חסרים).

> אם יש לך מקטעי קוד עם `` `` `` או `''` (נדיר), החרג אותם מההמרה — סרוק קודם
> `grep -n "\`\`\|''"` בתוך `lstlisting`.

```
l.NN \end{tikzpicture}
```
או — אם יש תוכן בין האיור לסוף הקובץ — ההודעה מצביעה על `\end{otherlanguage}` של
**טבלה רחוקה לגמרי**:
```
! \begin{tikzpicture} on input line K ended by \end{otherlanguage}
```
בשני המקרים **מספר השורה מטעה.** הסיבה: `tikzpic`/`qcirc` הם `\begin{otherlanguage}{english}`,
וכל חוסר-איזון בתוך האיור נבלע עד ה-`\end{otherlanguage}` הבא.

### השורש

`\begin{axis}` בלי `\end{axis}` תואם לפני `\end{tikzpicture}`. קל לשכוח כשכותבים הרבה
גרפים ברצף.

```latex
\begin{tikzpic}
\begin{tikzpicture}
\begin{axis}[...]
  \addplot{...};
\end{axis}            % ✅ חובה, גם אם יש רק \addplot אחד
\end{tikzpicture}
\end{tikzpic}
```

### האבחון

1. **`python3 scripts/check_bidi_figures.py *.tex`** — מדווח על השורה המדויקת של
   ה-`\begin{axis}` הלא-סגור (בניגוד לשורה המטעה של הקומפיילר). זה הכלי הראשון להריץ.
2. אם עדיין תקוע, קמפל את המודול הבודד בבידוד עם `-halt-on-error`, ואז חיתוך בינארי
   (`sed -n`/Python לחילוץ שורות — **לא** heredoc, שמחדיר תווי בקרה `^^H`).

### לא לבלבל — מה ש**אינו** הסיבה

חלוקה באפס ב-`\addplot` בנקודת קצה (למשל `tanh(sqrt((1-x)/x))` עם `domain=0:0.999`)
**אינה** הבעיה. תחת `compat=1.18` ‏`pgfplots` פשוט משמיט את נקודת הדגימה הסינגולרית
והעקומה יוצאת נכונה — אומת בבדיקה (הגרף תקין, 0 שגיאות). אל תבזבז זמן על זה; חפש את
ה-`\end{axis}` החסר.

> **למה זה חשוב:** התסמין (`Extra }`) זהה לחלוטין לבאגי-סוגריים אחרים, וההודעה מצביעה
> על שורה רחוקה. בלי לדעת את הדפוס הזה, מבזבזים זמן רב — לכן הבדיקה האוטומטית של איזון
> ה-`axis` חוסכת הכי הרבה.

## ★ אסור: שלישיית הבידוד במצב מתמטי

`\h{}`, `\en{}`, `\code{}`, `\texttt{}` — **כולם נשברים במצב מתמטי**:
`! Package fontspec Error: \rmfamily invalid in math mode` (או דומה). לעולם אל
תכניס אותם ל-`$...$`, `\[...\]`, `align*`.

| צורך | מה לעשות במתמטיקה |
| :---- | :---- |
| תווית עברית מתחת לסוגר (`\underbrace`) | `\underbrace{x}_{\text{\h{הזזת ראש}}}` — בתוך `\text{}` |
| קוד בתוך מתמטיקה | `\mathtt{...}` (לא `\texttt`/`\code`) |
| מזהה לטיני | פשוט כתוב אותו — `$x$`, `$rd$` (מתמטיקה ממילא LTR) |

```latex
% ✅ תווית עברית מתחת לסוגר, בתוך \text:
\[ T = \underbrace{T_{\text{seek}}}_{\text{\h{הזזת ראש}}} + \dots \]
```

(ראה גם כלל 4 ב-`recipe.md`: אל תכתוב עברית **ישירות** במתמטיקה — רק בתוך `\text{\h{}}`.)

## ★★ טבלאות: מסגרת אחידה + מספור דרך `tablebox`/`\tabcap` (לא `\caption*`)

> **תבניות מוכנות להעתקה:** לפני שבונים טבלה מאפס — פתחו את `assets/example_tables.tex`
> (מאגר בן 5 תבניות מוכחות: נתונים בסיסית, מקרא סמל←משמעות, עמודת קוד/LTR עם תא רב־שורתי,
> שני טורי פרוזה, וטבלה רחבה/טבלת אמת). כל תבנית מוערת ב״מתי להשתמש״; העתיקו את המתאימה
> והחליפו תוכן. זה מונע שחזור מאפס של המבנה (ואת המלכודות שלמטה). **הכלל המרכזי:**
> טבלה עברית־ראשית ⟵ `tabular` רגיל; טבלה LTR־ראשית (קוד/נוסחאות/טבלת־אמת) ⟵ עטיפה
> ב-`otherlanguage{english}` (מיישרת קוד לשמאל *וגם* מונעת היפוך סדר הטורים ע״י RTL).

**הבאג:** חבילת `caption` אינה נטענת. לכן `\caption*{...}` בתוך `table` **לא** מייצר כותרת
בלי מספר — LaTeX מפרש את ה-`*` כארגומנט, פולט שורה זרה ‎"טבלה N‏ : *"‎ ואז את הטקסט
בנפרד. התוצאה: כוכבית מסתורית + מספור לא-עקבי (טבלאות עם `\caption*` ממוספרות דרך מונה
`table`; טבלאות בלי כותרת — לא ממוספרות כלל). זה בדיוק חוסר-האחידות שצריך למנוע.

**הכלל:** *כל* טבלה במסמך עוברת דרך הסביבה `tablebox` + הפקודה `\tabcap`, **לעולם לא**
`\begin{table}`+`\caption*`. זה נותן מסגרת אחת אחידה (slate ניטרלי, נבדל מקופסאות הצבע
הסמנטיות), מספור אוטומטי (‎"טבלה N.M."‎) וכותרת *בתוך* המסגרת:
```
\begin{tablebox}\centering\small
  <tabular ...>                 % אפשר גם טקסט-מבוא לפני הטבלה, בתוך המסגרת
  \tabcap{<תיאור עברי>}          % ממוספר, בתוך המסגרת, בלי כוכבית, בלי caption*
\end{tablebox}
```
המנגנון מוגדר בפרמבל (מונה `tbl` מתאפס לפי `chapter`, `\thetbl=\thechapter.\arabic{tbl}`,
צבעי `cTblLine`/`cTblBg`). גם טבלה שאינה זקוקה לתיאור מקבלת `\tabcap` קצר — אחידות
(מסגרת+מספר לכולן) עדיפה על מספור חלקי. טבלה רחבה (7+ עמודות) נכנסת במסגרת; אם חורגת,
הקטן גופן ל-`\footnotesize` בתוך ה-`tablebox`.

## ★★ עמודת קוד בטבלה: כל הקוד LTR ומיושר-שמאל, בלי עברית מעורבת

**הבאג:** בטבלה עברית (RTL), עמודת `l` מיישרת תוכן LTR *צר* לימין ותוכן *רחב* לשמאל —
ערבוב מביך בין השורות. וגרוע יותר, דחיסת קוד עם הסבר עברי באותו תא
(`\code{bge ...} ; קוד S ; \code{skip:}`) יוצרת אי-בהירות מוחלטת: לא ברור איפה הקוד
נגמר ואיפה העברית מתחילה (המפרידים `;` מתהפכים בהקשר RTL, והאיים מתערבבים).

**הפתרון — שלושה חלקים:**
1. **עטוף את כל הטבלה ב-`\begin{otherlanguage}{english}`.** זה הופך אותה ל-LTR, ואז
   עמודות `l` מיישרות את *כל* הקוד לשמאל באופן אחיד (חד-שורתי ורב-שורתי כאחד). כותרות
   עבריות הופכות לאיים: `\multicolumn{1}{c}{\h{\textbf{מבנה ב-}\en{C}}}`.
2. **קוד רב-שורתי בתא ⇒ `tabular` מקונן מיושר-שמאל:**
   `\ttfamily\begin{tabular}[t]{@{}l@{}}line1\\line2\\line3\end{tabular}` — כל הוראה
   בשורה משלה, יישור-שמאל אחיד. (`\beginL`/`\endL` **אינם** מוגדרים ב-luatex — אל
   תשתמש בהם; `otherlanguage` הוא הדרך.)
3. **לעולם לא עברית בתוך רצף קוד.** מציין-מקום/הערה ⇒ הערת `#` *באנגלית*
   (`\# body (S)`), לא מילה עברית. כל הסבר עברי הולך ל-`\tabcap` או לעמודה עברית נפרדת.

**מבחן מהיר:** אם תא בעמודת-קוד מכיל גם `\code{}` *וגם* מילה עברית *וגם* `;`/`(` ביניהם —
זה ייקרא מבולבל. או שכולו קוד (עברית ל-caption), או שכולו פרוזה עברית עם איי-קוד משובצים
(תקין: ‎"`out` של הערך לרגיסטר `display`"‎ — העברית היא המארחת, לא להפך).

## ★★ חריגת שוליים (Overfull \hbox): איים רחבים שלא נשברים

**הסימן האובייקטיבי:** אזהרות `Overfull \hbox (Xpt too wide)` ב-log — אלה החיכוך הנמדד.
אסוף ומיין לפי חומרה:
```
grep -oE "Overfull .hbox \([0-9.]+pt too wide\)" LOG | sort -t'(' -k2 -rn
```
מעל ~5pt = חריגה **נראית** (טקסט תלוי מעבר לשוליים או מעבר לגבול הקופסה) — חובה לתקן.
מתחת ~3pt = גליף/פיסוק בודד נוגע בשוליים, תת-נראה — בד״כ מקובל ולא שווה ניסוח מאולץ.
מַפֵּה כל אזהרה למקור עם Python (קריאת השורה לפי מספרה; `cut -c` שובר תווי עברית).

**השורש (כמעט תמיד):** יחידת LTR **רחבה שאי-אפשר לשבור** בסוף או אמצע שורה עברית:
- אי מתמטי בודד רחב: `\en{$(ra,n{=}1),(ra,n{=}2),(ra,n{=}3)$}` הוא יחידה אטומית אחת.
- אי `\code{}` (monospace) — אינו נשבר גם בין מילים, ולכן בסוף שורה הוא חורג שמאלה.

**התיקונים (לפי מקרה — כולם נבדקו אמפירית):**
1. **רשימה בתוך אי מתמטי בודד → אי קומפקטי + ניסוח.** במקום שלושת הזוגות כאי רחב אחד:
   ‎"ערכי \en{n} הם \en{$3,2,1$}, וכל מסגרת שומרת \code{ra} ואת \en{n}"‎. (פיצול לשלושה
   איים `\en{$a$}, \en{$b$}, \en{$c$}` *כן* נשבר ושומר סדר 1,2,3 — אבל ה-linter מסמן
   `ADJACENT-LTR-ISLANDS`, ולכן עדיף אי בודד קומפקטי או ניסוח.)
2. **פריטים ממוספרים/מתויגים רצים בקופסה → שורה לכל פריט (`\\`).** `\textbf{(1)} … \\`
   `\textbf{(2)} … \\` וכו׳. זה מבודד את גלישת כל פריט (אי-קוד לא נדחף למיקום-שבירה גרוע
   בסוף שורה משותפת) **וגם** קריא וסַרִיק יותר. **זו הסיבה הנפוצה ביותר לחריגה** בקופסאות
   `keybox`/`notebox`/`thmbox` שמכילות איי קוד/מתמטיקה. אל תריץ צעדים ברצף.
3. **ריצת רשימת-קוד בקופסה → טבלת שני טורים** (תווית \textbar\ קוד, אחד לשורה),
   עטופה ב-`otherlanguage{english}` ליישור-שמאל אחיד ולפער עקבי גם לתוויות ארוכות
   (ראה סעיף ״עמודת קוד בטבלה״). עדיף על `p{}` עברי שדוחס תוויות ארוכות אל הקוד.
4. **`align*` דו-טורי רחב מדי → `{\small\begin{align*}…\end{align*}}`** (מתאים לכרטיס
   ייחוס צפוף), או הוצֵא את המשוואה הרחבה ביותר משורת הרשת לשורה נפרדת מתחת.
5. **משפט שמסתיים באי-`\code{}` רחב → קצר/נסח** כך שתיווצר הזדמנות-שבירה *לפני* האי
   (הסר מילות-קישור מיותרות, או הקדם את האי למקום עם מקום).
6. **`equation` יחידה רחבה ומרובת-איברים** (למשל מטריצת צפיפות עם 4 מכפלות טנזוריות)
   **→ פצל ל-`\begin{aligned}…\\…\end{aligned}`** בנקודת `+`/`=` טבעית, כך שהמשוואה
   נשברת לכמה שורות מיושרות במקום לחרוג. נצפה בבנייה אמת (192 עמ').

**רשת-ביטחון בפרמבל (גרסה 3.21):** `\emergencystretch=3em` מרפא אוטומטית את החריגות
ה**קטנות** של פרוזה עברית מיושרת (אי-LTR/מתמטי ליד השוליים בלי נקודת-שבירה חוקית) — הן
שורות tie-break שקשה לתקן בניסוח. הוא **לא** משפיע על חריגות של display-math / רשימות-קוד /
טבלאות — אלו עדיין מופיעות ב-log ודורשות את תיקוני-התוכן שלמעלה. (אומת: `emergencystretch`
אינו נוגע ב-overfull של `\[...\]` ולא משנה מספר-עמודים.)

**אחרי כל תיקון:** אסוף מחדש את רשימת ה-Overfull *וגם* אמת ויזואלית (`pdftoppm -r 150`+view)
שהאי כבר לא תלוי מעבר לשוליים/למסגרת. סדר-גודל טיפוסי: מסמך נקי = 0 חריגות נראות
(לכל היותר חריגות-פיסוק תת-3pt).

**מה חדש בגרסה 3.7** (מעל 3.4–3.6): טופל מחלקת באג חדשה — **pgfplots
`axis` בתוך אי-LTR**. `\begin{axis}` בלי `\end{axis}` תואם זורק `Extra }` שמתפשט ל-`\end{otherlanguage}`

**3.7 — תיקון מבני לבאג הסוגריים:** נמדד שההיקף רחב מסוגריים — `(`,`)`,`[`,`]`,**`$`** כולם מתהפכים בתוך `\texttt{}` בהקשר עברי (`$rs`→`rs$`, `arr[i]`→`[arr[i`, `(lw $t0,0($s0`). במקום לתפוס כל מקרה, ה-preamble מגדיר מחדש את `\texttt` כך שיבדד אוטומטית ל-LTR (עטוף `\ifmmode` לבטיחות מתמטיקה). מאז `\texttt{}` רגיל בטוח; `\code{}` הוא כינוי. ללא רגרסיה (`\texttt{עברית}` היה tofu גם קודם — בעיית גופן). בדיקת הלינטר ל-`\texttt`-עם-סוגר הוסרה כי המבנה מייתר אותה.
